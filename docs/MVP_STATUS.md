# MVP Status - July 2026

This document consolidates the original staged MVP plan, the three end-user/eval addendum notes, and the current implementation state after Sprint 020.

## North Star

The project is meant to prove this thesis:

> A small or cheap model becomes useful for complex planning if the harness owns global typed state and only feeds the model compiled, stage-local projections.

The proving-ground MVP is one-country travel planning, end to end:

```text
rambling prompt
  -> mission
  -> staged forms/questions
  -> candidate expansion
  -> choice collapse
  -> availability/quote preview
  -> approval-gated mocked hold/booking
  -> trip packet
  -> evidence trace
```

The UI must be testable by a real person. It is not enough for JSON fixtures to pass.

## What Exists

### Harness

- Typed trip/session state and many canonical boundaries exist.
- Prompt intake, mission proposal application, runtime action application, itinerary compilation, candidate selection, availability preview, edit-impact analysis, trip snapshot, replay audit, fidelity matrix, and live turn projection exist as isolated harness surfaces.
- `LiveSessionConductor` now accepts provider-agnostic live proposal envelopes and can apply or block mission proposals through canonical validation.
- Golden/deterministic traces exist and are useful for required offline regression.

### Providers

- OpenRouter-compatible model completion infrastructure exists with key loading, credit checks, timeout handling, typed provider exceptions, and no raw payload persistence.
- Provider-local runners/evaluators exist for mission planning, candidate expansion, hold prep, evidence export, fidelity, model roles, live preflight, and live mission proposal.
- `LiveMissionProposalRunner` can call a live model using strict JSON/schema output and returns sanitized accepted/blocked status rows.
- Sprint 020 attempted a guarded OpenRouter live mission proposal smoke but recorded `live_mission_proposal_provider_unavailable`, not an accepted live proposal.

### UI

- `/trip` is the end-user route and `/stage-cockpit` is the engineering route.
- The end-user UI has usable interaction primitives: composer/Ask affordance, assistant work bubbles, form cards, mood cards, selected-card user echo, planning timeline, task rail, provider status strip, and media hooks.
- The UI can show live/deterministic posture and fixed live proposal states.
- The UI currently still renders much of the visible planning interaction from deterministic/golden-trace state.

## What Is Still Synthetic

- The visible assistant cards are not yet primarily generated by live model output.
- The UI live proposal path uses bounded proposal packets and a harness application path, but the product session is not yet a complete live multi-turn session.
- The current live proposal slice is one mission proposal, not a full prompt -> question -> option -> selection -> availability -> approval-gate loop.
- Real availability/search is not the default end-user path.
- The Stage 6 fidelity matrix is still mostly deterministic and policy-driven rather than measured from live model runs.
- We do not yet have accepted live model traces proving a model can drive multiple safe turns through the harness.

## What Must Be Proven Next

Sprint 021 must prove that the harness can run a **multi-turn live interaction**, not merely a single proposal:

1. The user sends a prompt.
2. A real configured model receives a bounded harness packet.
3. The model returns strict structured output.
4. The harness validates, applies, or blocks the output.
5. The UI renders the resulting assistant card/work item.
6. The user selects or answers something.
7. A later model turn receives the updated projected state, not raw chat history.
8. The harness preserves state, provenance, approval gates, and no-mutation blocked paths.
9. Logs show exactly where a real model/provider run failed, without storing secrets or raw payloads.

This is the missing step between "one accepted proposal" and "expand beyond mission proposal."
